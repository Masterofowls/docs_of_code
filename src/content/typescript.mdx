<h2 id="types-and-narrowing">Types and Narrowing</h2>

```ts
type ApiResult =
  | { ok: true; data: { id: string; email: string } }
  | { ok: false; error: string };

function renderResult(result: ApiResult): string {
  return result.ok ? result.data.email : `error: ${result.error}`;
}
```

<h2 id="generics">Generics and Reuse</h2>

```ts
type Page<T> = {
  items: T[];
  nextCursor: string | null;
};

function firstItem<T>(page: Page<T>): T | null {
  return page.items[0] ?? null;
}
```

<h2 id="testing">Testing and Debugging</h2>

- Test runtime behavior and inferred types at boundaries.
- Keep compiler strict mode enabled in every package.
- Use source maps and inspect emitted JS for runtime mismatches.
- Fail CI on unused values and type errors.

<h2 id="common-errors">Common Error Patterns</h2>

- Undefined assignment errors: narrow or provide defaults.
- Widened literal types: use as const or satisfies where needed.
- Drift between runtime and static models: infer from one schema.

<h2 id="best-practices">Performance and Security</h2>

- Types improve correctness, not trust boundaries; validate input.
- Prefer simple and readable types over deeply nested type meta code.
- Split types by feature boundaries to keep compile speed stable.
